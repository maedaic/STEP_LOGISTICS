# OCR-001 調査メモ：OCR結果と画像内容の差異

priority: Highest / status: Investigation

## 1. 現状（重要）

現在のアプリの「OCR」は **スタブ**です（`js/app.js` の `runOcrStub()`）。実際の画像解析は行わず、
アップロードのたびに固定のサンプル明細（伝票ごとに別セット）を返しています。

> つまり現時点の「OCR結果と実際の伝票が一致しない」の主因は、**実OCRが未接続**であることです。
> 本チケットは実OCRを載せる前提での精度設計・調査を目的とします。

## 2. 方針（OCR単体で完璧を目指さない）

伝票OCRは「印字/手書き混在・傾き・影・かすれ」があり、OCR単体の文字認識だけで高精度化するのは高コスト。
**「OCR候補 × 商品マスター照合」で最終精度を上げる**方針が現実的（本チケットの objective と一致）。

```
画像 → 前処理 → OCR(候補複数) → 正規化 → 商品マスター照合(あいまい一致) → 確認画面で人が最終確定
```

## 3. OCRエンジン候補

| 方式 | 長所 | 短所 | 備考 |
|---|---|---|---|
| Tesseract.js（WASM・オフライン） | 無料・ローカル完結・PWA適合 | 日本語/手書き精度が低め・前処理必須 | まず検証に最適 |
| Google Cloud Vision / Document AI | 高精度・表構造抽出可 | 通信必須・従量課金・情報を外部送信 | 精度重視ならこれ |
| Azure AI Vision (Read) | 高精度・帳票向け | 同上 | 同上 |
| 専用フォーマット化（帳票テンプレ） | 最も安定 | 伝票様式の統一が前提 | 中期の本命 |

推奨: **短期=Tesseract.jsで前処理込み検証 → 精度不足なら Document AI 等のクラウドへ**。
（クラウドは伝票画像を外部送信するため、社内データの取り扱い可否を要確認）

## 4. 前処理（精度への寄与が大きい順）

1. **二値化＋コントラスト補正**（大津の二値化 / adaptive threshold）
2. **傾き補正（回転）** — テキスト行の角度推定で水平化
3. **台形補正（射影変換）** — 斜め撮影の矩形化（OpenCV.js の getPerspectiveTransform）
4. 解像度正規化・ノイズ除去（メディアンフィルタ）
5. 品番列/数量列の**ROI切り出し**（列位置が決まっている伝票なら劇的に安定）

## 5. 商品マスター照合ロジック（アプリ側で実装済み/拡張予定）

- **正規化（実装済み: `normalizeCode()` / `resolveProductCode()`）**
  大小文字・全半角・空白・ハイフン有無を吸収（`c6`→`C6`, `Ｃ６`→`C6`, `c-6`→`C6`）。
- **あいまい一致（拡張予定）**: レーベンシュタイン距離で近い品番を候補化
  （例: `C8N`↔`C6N` の誤認を距離1で候補表示）。
- **接頭辞グルーピング**: `C6` / `C6-12` / `C6D` を系統でまとめ、数量列と突き合わせて妥当性チェック。
- **数量の健全性チェック**: 桁化け（`l`→`1`, `O`→`0`）の補正、在庫上限との比較で異常値を警告。

## 6. 今後（future 準拠）

- **OCR確認画面**: OCR直後に「品番・数量」を表形式で提示し、確定前に人がレビュー。
- **認識候補表示**: 各セルにマスター照合の上位候補をドロップダウン表示。
- **ワンクリック修正**: 候補タップで確定 → 伝票明細へ反映（現行の「伝票単位表示」にそのまま接続可能）。

## 7. 次アクション（提案）

1. Tesseract.js を1画面で試作し、実伝票10枚で認識率をベースライン測定。
2. 前処理（二値化→傾き補正）を入れて再測定、寄与を定量化。
3. `resolveProductCode()` にレーベンシュタイン候補を追加し、照合後精度を測定。
4. 目標精度に届かなければ Document AI 等の帳票OCRへ切替を検討。

> OCR確認画面と candidate 表示にそのまま流用できる構造になっています。
